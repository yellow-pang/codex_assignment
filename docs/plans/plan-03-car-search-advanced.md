# 3단계 차량 검색 고도화 작업 계획

## 1. 작업 배경

`docs/실시간_Car_Market_향후_개발_계획서.md`의 3단계 항목에 따라 차량 검색 기능을 요구사항 기준으로 확장한다.

현재 서버는 MongoDB Atlas `cars` 컬렉션 기반 CRUD로 전환되어 있지만, 검색 기능은 아직 아래처럼 분리되어 있다.

| 현재 기능   | 현재 API                                           | 한계                               |
| ----------- | -------------------------------------------------- | ---------------------------------- |
| 제조사 검색 | `GET /api/cars/search?company=HYUNDAI`             | 제조사 단일 조건만 지원            |
| 가격 필터   | `GET /api/cars/filter?minPrice=1000&maxPrice=3000` | 검색 API와 분리되어 복합 검색 불가 |

신규 요구사항은 `GET /api/cars/search` 하나에서 차량명, 제조사, 가격 범위, 연식 범위를 함께 사용할 수 있어야 한다.

## 2. 작업 목표

- `GET /api/cars/search`를 복합 검색 API로 확장한다.
- `keyword`, `company`, `minPrice`, `maxPrice`, `minYear`, `maxYear` 쿼리 파라미터를 지원한다.
- 기존 `/api/cars/filter`는 빌드와 검증 과정에서 문제가 없으면 제거한다.
- 프론트엔드 목록 화면의 검색 폼에 차량명, 제조사, 가격, 연식 조건을 모두 추가한다.
- 검색과 초기화 동작을 한 화면에서 이해하기 쉽게 정리한다.

## 3. 백엔드 구현 계획

### 3.1 검색 쿼리 생성 함수 분리

`server.js`에 검색 조건을 MongoDB query로 바꾸는 함수를 추가한다.

예상 함수:

```js
function createCarSearchQuery(queryParams) {
  const query = {};
  // keyword, company, price, year 조건 구성
  return query;
}
```

검색 조건 처리 기준:

| 파라미터   | 처리 방식                                               |
| ---------- | ------------------------------------------------------- |
| `keyword`  | `name` 필드에 대소문자 구분 없는 부분 검색 정규식 적용  |
| `company`  | 공백 제거 후 대문자로 변환해 `company` 필드와 일치 검색 |
| `minPrice` | 숫자로 변환 후 `price.$gte` 적용                        |
| `maxPrice` | 숫자로 변환 후 `price.$lte` 적용                        |
| `minYear`  | 숫자로 변환 후 `year.$gte` 적용                         |
| `maxYear`  | 숫자로 변환 후 `year.$lte` 적용                         |

### 3.2 잘못된 숫자 조건 처리

가격과 연식 조건은 먼저 공백, 탭, 줄바꿈을 제거한 뒤 숫자로 검증한다. 숫자가 아닌 값이 들어오면 `400` 상태 코드와 안내 메시지를 응답한다.

예시:

```json
{ "message": "가격과 연식 검색 조건은 숫자로 입력해야 합니다." }
```

### 3.3 검색 API 통합

`GET /api/cars/search`에서 새 검색 쿼리 생성 함수를 사용한다.

예시:

```text
GET /api/cars/search?keyword=sonata&company=HYUNDAI&minPrice=1000&maxPrice=3000&minYear=2020
```

MongoDB 검색 예시:

```js
{
  name: { $regex: "sonata", $options: "i" },
  company: "HYUNDAI",
  price: { $gte: 1000, $lte: 3000 },
  year: { $gte: 2020 }
}
```

### 3.4 `/api/cars/filter` 제거

사용자 승인에 따라 `/api/cars/filter` 호환 라우트는 유지하지 않는다.

처리 방향:

- 새 프론트엔드는 `/api/cars/search`를 사용한다.
- 빌드와 검증 과정에서 기존 `/api/cars/filter` 의존 코드가 없는지 확인한다.
- 문서에는 기존 가격 필터 API가 복합 검색 API로 통합되었음을 기록한다.

## 4. 프론트엔드 구현 계획

### 4.1 검색 상태 확장

`frontend/src/App.jsx`의 `filters` 상태를 아래 조건까지 확장한다.

```js
{
  keyword: "",
  company: "",
  minPrice: "",
  maxPrice: "",
  minYear: "",
  maxYear: ""
}
```

### 4.2 검색 폼 확장

목록 화면 검색 영역에 다음 입력값을 추가한다.

| 항목      | 입력 UI     |
| --------- | ----------- |
| 차량명    | 텍스트 입력 |
| 제조사    | 텍스트 입력 |
| 최소 가격 | 숫자 입력   |
| 최대 가격 | 숫자 입력   |
| 최소 연식 | 숫자 입력   |
| 최대 연식 | 숫자 입력   |

현재 UI는 daisyUI 기반이므로 이번 단계에서는 기존 화면 톤을 크게 바꾸지 않고, 검색 필드만 요구사항에 맞게 확장한다.

### 4.3 검색/초기화 동작

- `검색` 버튼은 입력된 조건을 `URLSearchParams`로 묶어 `/api/cars/search`를 호출한다.
- 조건이 없으면 `/api/cars` 전체 목록을 조회한다.
- `초기화` 버튼은 검색 조건을 비우고 전체 목록을 다시 조회한다.
- 기존 `제조사 검색`, `가격 필터`처럼 분리된 버튼은 하나의 검색 흐름으로 통합한다.
- 가격과 연식 입력은 `type="number"`와 적절한 안내 문구로 숫자 입력을 유도한다.
- 프론트엔드에서도 trim 후 숫자가 아닌 값이면 API 요청 전에 오류 메시지를 보여주고 입력을 다시 유도한다.

## 6. 검증 계획

가능한 범위에서 아래 검증을 진행한다.

| 검증 항목      | 명령 또는 방법                                           |
| -------------- | -------------------------------------------------------- |
| 서버 문법 확인 | `node --check server.js`                                 |
| 루트 빌드      | `npm.cmd run build`                                      |
| 서버 실행      | `npm.cmd start`                                          |
| 검색 API 확인  | 실제 `MONGODB_URI`가 있을 때 `/api/cars/search?...` 호출 |

현재 로컬 환경에 실제 `MONGODB_URI`가 없다면 서버 실행은 MongoDB 연결 실패 메시지까지만 확인하고, 실제 API 호출 검증은 사용자가 환경변수를 등록한 뒤 진행한다.

## 7. 이번 단계에서 하지 않을 일

- MongoDB 컬렉션 구조를 크게 바꾸지 않는다.
- 기존 차량 등록 데이터 구조에 사진, 연료, 주행거리, 지역, 설명 필드를 추가하지 않는다.
- `multer` 사진 업로드는 4단계에서 진행한다.
- Firebase 인증과 딜러 권한 체크는 5단계에서 진행한다.
- daisyUI 제거 또는 전체 목록 카드형 UI 개편은 별도 UI 단계에서 진행한다.
- `/cars` 레거시 호환 라우트는 이번 단계에서 제거하지 않는다.

## 8. 사용자 확인 필요 사항

구현 전에 아래 방향을 확인받는다.

1. 제조사 입력 방식은 이번 단계에서 기존처럼 자유 텍스트 입력으로 유지하고, 이후 UI 개편 때 select로 바꾼다.
2. `/api/cars/filter`는 호환 라우트로 유지하지 않고 제거한다.
3. 검색 결과가 없을 때는 오류가 아니라 빈 배열을 보여주고, 프론트엔드에는 "검색 결과가 없습니다." 안내를 표시한다.
4. 숫자가 아닌 가격/연식 조건은 프론트엔드에서 먼저 입력을 다시 유도하고, API로 들어온 경우 서버에서 `400` 오류로 처리한다.

위 네 가지 방향은 사용자 승인 완료 상태이며, 이 계획에 따라 코드 구현을 진행한다.
