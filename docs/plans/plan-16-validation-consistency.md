# 16단계 입력 검증, 정합성, 동시성, 중복 요청 방지 계획

## 1. 문서 목적

15단계에서 Firebase ID Token 기반 서버 인증과 권한 검증을 도입했다.
이번 16단계는 인증된 사용자라도 잘못된 값, 과도한 입력, 반복 클릭, 짧은 시간 내 중복 요청, 동시 메시지 전송이 들어올 때 서버와 화면이 안정적으로 동작하도록 보강하는 계획을 정리한다.

이 문서는 코드 수정 전 사용자 확인을 받기 위한 계획 문서다.
코드 수정은 아래 확인 필요 항목에 대한 사용자 승인 후 진행한다.

## 5. 사용자 확인이 필요한 항목

| 확인 항목                                      | 추천 방향                                                                                                                      | 이유                                                                              | 영향                                                                    |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| MongoDB index 또는 unique index 실제 생성 여부 | 코드에 `ensureIndexes` 함수를 추가하되 기본 index만 안전하게 생성, unique index는 `users.uid`, `chat_rooms.roomId`만 우선 적용 | 중복 사용자/상담방을 DB 수준에서 막을 수 있고 기존 컬렉션 구조를 크게 바꾸지 않음 | 기존 DB에 중복 데이터가 있으면 unique index 생성 실패 가능              |
| DB 컬렉션 구조나 필드 정책 변경 여부           | 큰 구조 변경 없이 기존 필드 유지, 필요하면 `messages.clientMessageId` 같은 선택 필드만 추가                                    | 컬렉션 구조 대규모 변경을 피하면서 중복 메시지 방어에 도움                        | 새 선택 필드 추가 시 문서화 필요                                        |
| 중복 요청 방어 기준                            | 1차는 서버 메모리 TTL 기준으로 구현, DB unique는 상담방/사용자처럼 명확한 중복에만 적용                                        | Render 단일 인스턴스 과제 환경에서는 단순하고 영향이 작음                         | 서버 재시작 또는 다중 인스턴스에서는 메모리 중복 방어가 공유되지 않음   |
| 프론트 UX 버튼/상태 표시 범위                  | 대규모 UI 변경 없이 제출 버튼 disable, 로딩 문구, 메시지 전송 중 상태만 추가                                                   | 따닥 방지 목적을 달성하면서 화면 구조를 크게 바꾸지 않음                          | 버튼 문구와 disabled 상태 일부 변경                                     |
| 업로드 제한값 변경 여부                        | 기존 5MB, jpg/jpeg/png/webp 유지                                                                                               | 이미 문서와 UI에 5MB 기준이 정리되어 있고 과제 범위에 적합                        | 제한값을 바꾸지 않으면 문서 영향 최소                                   |
| 메시지 중복 방어 방식                          | 프론트에서 전송 중 버튼 비활성화 + 서버에서 동일 사용자/방/본문의 짧은 TTL 중복 차단                                           | 사용자가 빠르게 두 번 누르는 일반 문제를 줄일 수 있음                             | 같은 문장을 짧은 시간 안에 의도적으로 반복 전송하는 경우 차단될 수 있음 |
| 메시지 정렬 기준                               | `createdAt` 오름차순 + `_id` 오름차순 정렬로 문서화하고 구현                                                                   | 같은 밀리초에 저장된 메시지의 표시 순서를 안정화                                  | 기존 메시지 데이터와 호환                                               |

## 6. 구현 목표

1. 차량 등록/수정 입력값의 서버 최종 검증을 강화한다.
2. 사용자 프로필 저장과 역할 변경 입력값의 서버 최종 검증을 강화한다.
3. 채팅 메시지 길이, 공백, 중복 전송 검증을 강화한다.
4. 차량 등록/수정/삭제, 상담방 생성, 관리자 역할 변경, 메시지 전송 버튼에 loading/disabled 처리를 추가한다.
5. 백엔드에서 짧은 시간 내 동일 요청을 방어하는 작은 중복 요청 guard를 도입한다.
6. 상담방 생성 upsert 정책을 점검하고 필요한 index 또는 unique index 후보를 정리한다.
7. 메시지 동시 전송 시 정렬 기준을 `createdAt`, `_id` 기준으로 정리한다.
8. 업로드 보안 설정을 재점검하고 파일 제한 기준을 문서화한다.
9. 프론트 1차 검증과 백엔드 최종 검증의 책임 분리를 문서화한다.

## 7. 백엔드 변경 계획

### 7.1 차량 입력 검증

추천 구현 위치:

| 파일                               | 계획                                                                |
| ---------------------------------- | ------------------------------------------------------------------- |
| `backend/utils/normalizers.js`     | `validateCarInput`, 허용 제조사/차종/연료 목록, 길이 제한 상수 추가 |
| `backend/services/cars.service.js` | `createCar`, `updateCar`에서 정규화 후 검증 실패 시 `400` 응답      |

추천 검증 기준:

| 필드          | 기준                                      |
| ------------- | ----------------------------------------- |
| `name`        | 필수, 2~80자                              |
| `company`     | 필수, 허용 제조사 목록 또는 2~40자 대문자 |
| `year`        | 정수, 1900 이상 현재 연도 이하            |
| `price`       | 숫자, 0보다 큼, 과도한 값 제한            |
| `type`        | 필수, 허용 차종 목록                      |
| `fuel`        | 필수, 허용 연료 목록                      |
| `mileage`     | 숫자, 0 이상                              |
| `location`    | 필수, 2~40자                              |
| `description` | 필수, 10~1000자                           |

프론트 검증은 사용자에게 빠르게 알려주는 1차 검증이고, 서버 검증은 API 도구로 직접 호출해도 반드시 적용되는 최종 검증으로 둔다.

### 7.2 사용자 프로필과 역할 변경 검증

추천 구현 위치:

| 파일                                | 계획                                                     |
| ----------------------------------- | -------------------------------------------------------- |
| `backend/utils/normalizers.js`      | `validateUserInput` 보강, `validateRoleUpdateInput` 추가 |
| `backend/services/users.service.js` | 역할 변경 조합 검증 강화                                 |

검증 기준:

- `uid`, `email`은 Firebase 토큰 기준을 우선 사용한다.
- `displayName`은 필수, 2~40자 권장.
- `role`은 `buyer`, `dealer`, `admin`만 허용한다.
- `dealerStatus`는 `none`, `pending`, `approved`, `rejected`만 허용한다.
- `role: dealer`는 `dealerStatus: approved`와 함께 저장한다.
- `role: admin`은 `dealerStatus: none`으로 정리한다.
- 자기 자신의 admin 권한 해제 방어는 유지한다.

관리자 권한 정책 자체를 바꾸지는 않는다.

### 7.3 채팅 메시지 검증

추천 구현 위치:

| 파일                                | 계획                                            |
| ----------------------------------- | ----------------------------------------------- |
| `backend/utils/normalizers.js`      | `validateChatMessageText` 추가                  |
| `backend/services/chats.service.js` | 메시지 저장 전 길이와 공백 검증, 정렬 기준 보강 |
| `backend/sockets/chat.socket.js`    | 중복 전송 오류를 `chat-error`로 사용자에게 전달 |

검증 기준:

- trim 후 빈 문자열이면 저장하지 않는다.
- 최대 길이는 기존 1000자를 유지한다.
- 1000자를 초과하면 조용히 자르지 않고 오류로 반환하는 방식을 추천한다.
- 동일 상담방, 동일 사용자, 동일 본문이 짧은 시간 안에 반복되면 중복 요청으로 차단한다.

### 7.4 중복 요청과 따닥 요청 방어

추천 구현:

```text
backend/utils/requestGuard.js
```

역할:

- 사용자 UID, 작업명, 대상 ID, 본문 hash를 조합해 key를 만든다.
- TTL Map으로 짧은 시간 동안 같은 요청을 차단한다.
- 차량 등록/수정/삭제, 상담방 생성, 메시지 전송에 우선 적용한다.

추천 TTL:

| 작업                | TTL   |
| ------------------- | ----- |
| 차량 등록/수정/삭제 | 3초   |
| 상담방 생성         | 3초   |
| 메시지 전송         | 1.5초 |
| 관리자 역할 변경    | 2초   |

주의:

- 이 방식은 서버 메모리 기준이므로 단일 Render Web Service에 적합하다.
- 서버 재시작 또는 다중 인스턴스에서는 공유되지 않는다.
- 운영 수준으로 확장하려면 DB unique key, idempotency key, Redis 같은 별도 저장소가 필요하다.

### 7.5 상담방 생성 upsert 정책

현재 상담방은 `carId`, 인증 buyer UID, 차량 dealer UID로 만든 `roomId`를 기준으로 `findOneAndUpdate(..., { upsert: true })` 한다.

추천 유지:

- 같은 구매자, 같은 딜러, 같은 차량이면 기존 상담방을 재사용한다.
- 상담방 생성 요청이 반복되어도 방이 늘어나지 않도록 한다.
- `updatedAt`은 재진입 시 갱신될 수 있다.

검토 대상:

- `chat_rooms.roomId` unique index를 실제 생성할지 확인 필요.
- 기존 DB에 중복 `roomId`가 있으면 정리 없이 unique index 생성이 실패할 수 있다.

### 7.6 메시지 동시 전송 정렬 기준

추천 기준:

- 조회 정렬: `{ createdAt: 1, _id: 1 }`
- 상담방 목록 정렬: `{ updatedAt: -1, _id: -1 }`
- 차량 목록/검색 정렬: 기존 `{ createdAt: -1, _id: -1 }` 유지

`createdAt`이 같은 밀리초로 기록될 수 있으므로 `_id`를 보조 정렬 기준으로 둔다.

### 7.7 MongoDB index 후보

사용자 확인 후 실제 생성 여부를 결정한다.

| 컬렉션       | index 후보                            | unique         | 추천 여부 | 이유                                                           |
| ------------ | ------------------------------------- | -------------- | --------- | -------------------------------------------------------------- |
| `users`      | `{ uid: 1 }`                          | 예             | 추천      | Firebase UID 중복 사용자 방지                                  |
| `users`      | `{ email: 1 }`                        | 아니오 또는 예 | 보류      | Firebase가 이메일 중복을 관리하지만 기존 데이터 상태 확인 필요 |
| `chat_rooms` | `{ roomId: 1 }`                       | 예             | 추천      | 상담방 upsert 중복 방지                                        |
| `chat_rooms` | `{ buyerId: 1, updatedAt: -1 }`       | 아니오         | 추천      | 내 상담방 목록 조회 성능                                       |
| `chat_rooms` | `{ dealerId: 1, updatedAt: -1 }`      | 아니오         | 추천      | 딜러 상담방 목록 조회 성능                                     |
| `messages`   | `{ roomId: 1, createdAt: 1, _id: 1 }` | 아니오         | 추천      | 이전 메시지 조회와 정렬 안정화                                 |
| `cars`       | `{ createdAt: -1, _id: -1 }`          | 아니오         | 선택      | 목록 정렬 성능                                                 |
| `cars`       | `{ company: 1, year: -1, price: 1 }`  | 아니오         | 선택      | 검색 조건이 많아질 때 성능 보조                                |

unique index는 기존 데이터에 중복이 없는지 확인한 뒤 적용해야 한다.

### 7.8 업로드 보안 추가 점검

현재 유지 기준:

- 최대 5MB
- 확장자: `.jpg`, `.jpeg`, `.png`, `.webp`
- MIME: `image/jpeg`, `image/png`, `image/webp`
- 저장 파일명은 서버 생성 랜덤 파일명 사용

추가 점검 후보:

- 파일 필드 개수는 `image` 1개만 허용한다.
- 업로드 실패 시 DB 저장을 진행하지 않는다.
- 파일명에 사용자 입력을 사용하지 않는 현재 정책을 문서화한다.
- 실제 이미지 내용 검사까지는 새 패키지 또는 추가 처리 비용이 있어 이번 단계에서는 보류한다.

## 8. 프론트엔드 변경 계획

### 8.1 버튼 loading/disabled 처리

추천 적용 대상:

| 파일                                             | 대상                                                      |
| ------------------------------------------------ | --------------------------------------------------------- |
| `frontend/src/App.jsx`                           | 검색, 초기화, 차량 등록/수정/삭제, 상담방 생성, 딜러 신청 |
| `frontend/src/components/CarForm.jsx`            | 등록/수정 저장 버튼에 `isSubmitting` prop 적용            |
| `frontend/src/components/DeleteConfirmModal.jsx` | 삭제 확인 버튼 disable                                    |
| `frontend/src/components/AdminUserPanel.jsx`     | 역할 변경 버튼별 처리 중 상태                             |
| `frontend/src/components/ChatRoom.jsx`           | 메시지 전송 중 버튼 disable, 긴 메시지 1차 차단           |

버튼 문구 예시:

- `등록 중...`
- `수정 중...`
- `삭제 중...`
- `상담방 준비 중...`
- `전송 중...`

대규모 UI 방향은 바꾸지 않고 기존 Tailwind class 체계를 유지한다.

### 8.2 프론트 1차 검증

프론트는 사용자가 입력 즉시 이해할 수 있도록 빠른 안내를 담당한다.

- 차량 폼에서 필수값, 숫자 범위, 설명 길이를 먼저 확인한다.
- 메시지는 공백만 있으면 전송하지 않는다.
- 메시지 길이가 1000자를 넘으면 전송 전 안내한다.
- 중복 클릭 방지를 위해 요청 진행 중 버튼을 비활성화한다.

백엔드 최종 검증과 기준이 달라지지 않도록 상수나 문서 기준을 맞춘다.

## 10. 작업 순서

1. 사용자 확인 항목에 대한 승인 내용을 확정한다.
2. 차량/사용자/메시지 검증 기준을 `backend/utils/normalizers.js`에 추가한다.
3. 서비스 레이어에서 검증 실패를 `400` 오류로 반환하도록 연결한다.
4. 서버 메모리 TTL 기반 중복 요청 guard를 추가한다.
5. 차량 등록/수정/삭제, 상담방 생성, 메시지 전송, 역할 변경에 guard를 적용한다.
6. 상담방 upsert와 메시지 정렬 기준을 보강한다.
7. 사용자 승인 시 MongoDB index 생성 helper를 추가하고 서버 시작 시 실행한다.
8. 업로드 설정을 추가 점검하고 문서와 오류 메시지를 정리한다.
9. 프론트 버튼 loading/disabled 상태를 적용한다.
10. 프론트 1차 검증 메시지를 백엔드 최종 검증 기준과 맞춘다.
11. 구현 완료 문서 `docs/steps/2026-06-07-16-validation-consistency.md`를 작성한다.
12. PR 요약 문서 `docs/pr/2026-06-07-16-validation-consistency-pr.md`를 작성한다.
13. `docs/progress.md`에 16단계 결과를 기록한다.

## 12. 검증 기준

| 검증           | 기준                                                                |
| -------------- | ------------------------------------------------------------------- |
| 서버 문법 확인 | 수정한 `backend/**/*.js`에 대해 `node --check` 성공                 |
| 프론트 빌드    | `npm.cmd --prefix frontend run build` 성공                          |

| 사용자 검증    | 잘못된 역할/딜러 상태 조합이 `400` 응답                             |
| 메시지 검증    | 공백 메시지, 1000자 초과 메시지, 짧은 시간 중복 메시지가 차단       |
| 중복 제출      | 차량 등록/수정/삭제, 상담방 생성, 역할 변경 버튼이 요청 중 비활성화 |

| 메시지 정렬    | 같은 방 메시지가 `createdAt`, `_id` 기준으로 안정 정렬              |
| 업로드 검증    | 5MB 초과 또는 허용되지 않은 확장자/MIME은 `400` 응답                |
| 문서 검증      | Step/PR/progress에 프론트 1차 검증과 백엔드 최종 검증 기준 반영     |

실제 MongoDB unique index 생성과 API 실동작 검증은 로컬 `.env`의 MongoDB/Firebase 설정 준비 여부에 따라 제한될 수 있다.

## 13. 코드 수정 전 확인 질문

아래 방향으로 진행해도 되는지 확인이 필요하다.

1. MongoDB index 생성 helper를 추가하고, `users.uid`, `chat_rooms.roomId` unique index와 메시지/상담방 조회용 일반 index를 실제로 생성해도 되는가?
2. DB 컬렉션 구조는 유지하되, 메시지 중복 방어가 필요할 경우 `messages.clientMessageId` 같은 선택 필드를 추가할 수 있게 열어두어도 되는가? 추천은 이번 1차에서는 필드 추가 없이 서버 메모리 TTL 방식이다.
3. 중복 요청 방어는 서버 메모리 TTL 기준으로 우선 구현해도 되는가? 추천은 단일 Render Web Service에 맞는 메모리 방식이다.
4. 프론트 UX는 버튼 disable, 로딩 문구, 전송 중 상태 표시까지만 바꾸고 화면 구조는 유지해도 되는가?
5. 업로드 제한값은 기존 5MB와 jpg/jpeg/png/webp 허용 정책을 유지해도 되는가?
6. 메시지 1000자 초과 입력은 서버에서 잘라 저장하지 않고 오류로 차단하는 방식으로 바꿔도 되는가?
